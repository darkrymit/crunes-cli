# AGENTS.md

## Mandatory Order of Operations

Before brainstorming, planning, or touching any code:

1. **Get live codebase context via context runes** тАФ scope `m` to the modules relevant to the task and `kb` to matching entries; avoid loading all modules unless the task spans the full codebase. Re-run crunes at any point if exploration reveals additional dependencies.
2. **Then brainstorm, plan, and code** тАФ in that order.

## Restrictions

- **Never publish to npm directly** тАФ publishing is handled by CI on tag push. Only bump the version, update the changelog, commit, tag, and push.
- **Never modify `dist/`** тАФ it is generated by `npm run build` (esbuild) and committed only as part of a release. Do not hand-edit it.
- **Never run `npm publish`** unless the user explicitly asks.
- **Always sync `package-lock.json`** тАФ after bumping the version in `package.json`, run `npm install` to regenerate the lockfile before committing. The `release` rune will flag a mismatch with a тЪа indicator.
- **Rune files are ESM** тАФ all files under `.crunes/runes/` and `src/` use `import`/`export`. Never use `require()`.
- **Local runes run inside `isolated-vm`** тАФ they cannot access Node builtins directly. All I/O goes through `utils.fs`, `utils.shell`, `utils.json`, `utils.fetch`, `utils.env`, and `utils.archive`.
- **Test before committing rune changes** тАФ run `node dist/cli.js use <rune> --plain` to verify output before staging.
- **Fetch module context via `crunes use m=<module>`** for the module(s) you are about to touch тАФ fall back to the module's `README.md` only if `crunes` is unavailable.
- **Prefer `crunes` over `Glob`/`Grep`/`ls`** тАФ use filesystem tools only when crunes cannot answer the question.
- **Only read files that will change what you implement** тАФ ask "will this result change my approach?" before reading anything.

## Architecture Overview

**Stack**: Node.js тЙе 20 тАв ESM тАв esbuild (bundler) тАв commander (CLI) тАв isolated-vm (plugin sandboxing) тАв vitest (tests)

**Entry point**: `src/cli/cli.js` тАФ process bootstrap, Node snapshot workarounds, and `-v` flag disambiguation.

**Core execution path**: `crunes use <key>` тЖТ `src/rune/commands/use.js` тЖТ `src/rune/resolver.js` (`runRune`) тЖТ resolves local vs. plugin тЖТ `src/rune/isolation/runner.js` (isolated-vm) тЖТ returns `Section[]` тЖТ `src/shared/render.js` тЖТ stdout.

**Key resolution order** (for a bare key with no prefix):
1. Project config (`.crunes/config.json` тЖТ `runes.<key>.path`)
2. Enabled plugins (auto-resolved; errors if ambiguous across multiple plugins)

**Prefixes**: `local:<key>` forces project config only. `<plugin>:<key>` forces a specific plugin.

### Module Map

Prefer `crunes use m` тАФ this is a static fallback only.

- `cli` тАФ Entry point (`cli.js`), Commander setup (`program.js`), `-v` disambiguation, general CLI commands.
- `core` тАФ `loadConfig()`, `CircularRuneError`, shared config types.
- `marketplace` тАФ Marketplace source URL management, plugin index browsing and caching.
- `plugin` тАФ Plugin registry, install, consent, dep resolution, store paths.
- `rune` тАФ Key resolution, sandboxed VM execution, utils API, permissions.
- `shared` тАФ `render.js`, `output.js` тАФ cross-cutting utilities with no domain coupling.
- `template` тАФ Rune template listing and scaffolding.

### Module Documentation

Every module has a `README.md` stub at `src/<module>/README.md` (file index, submodules, related modules). **Do not put narrative docs in READMEs** тАФ narrative belongs in the KB.

KB vault: `docs/knowledge-base/` тАФ module notes at `modules/<module>.md`, flows at `flows/<flow>.md`, system at `system/`.

- **Single module:** `crunes use m=<module> -a kb=m,<module>`
- **Submodule:** `crunes use m=<module>.<submodule>` (dot = submodule path separator, e.g. `m=rune.isolation`)
- **Multiple modules:** `crunes use m=<mod1>,<mod2> -a kb=m,<mod1>,<mod2>`
- **Priority order:** chain `-a` groups so the most task-relevant module comes first тАФ e.g. `crunes use m=rune.isolation -a kb=m,rune.isolation -a m=plugin -a kb=m,plugin`

Include `kb` alongside `m` when design decisions or subtle gotchas are likely to matter тАФ new features, cross-module contracts, anything non-obvious from source. Skip for purely structural reads.

### Context Runes

- `release` тАФ Current version, lockfile sync status, branch/tag, last 10 commits, recent changelog, release process steps.
- `m` тАФ Module structure tree + file tree. No args: all modules. `m=rune` тЖТ rune module. `m=rune.isolation` тЖТ submodule (dot = path separator). `m=plugin,rune` тЖТ multiple modules. Sections: `layout` (submodule tree), `files` (JS file tree), `readme`.
- `kb` тАФ Knowledge base notes. No args: index of all entries. `kb=m,rune,plugin` тЖТ specific module entries. `kb=f,use` тЖТ flow doc. Sections: one per entry name.

**Fallback** (crunes unavailable): Module Map above for `m`; `docs/knowledge-base/index.md` for `kb`.

## Release Process

1. Bump `version` in `package.json`
2. Run `npm install` to sync `package-lock.json`
3. Add `## [x.y.z] - YYYY-MM-DD` entry to `CHANGELOG.md`
4. Commit: `git commit -m "chore: release vX.Y.Z"`
5. Tag: `git tag vX.Y.Z`
6. Push: `git push origin main --tags` тАФ publish CI fires automatically on the tag

Verify before pushing: `node dist/cli.js use release --plain` should show matching versions, `тЬУ` on lockfile, and `0 (in sync)` on unpushed.

## Testing Philosophy

- Test **API contracts**, not implementation details.
- Use **prose assertions** for distinct behavioural contracts (error types, option semantics, edge cases).
- Use **snapshots** for scaffold/template string outputs тАФ prose tests guard logic, snapshots guard exact formatting.
- Avoid testing **trivial wrappers** тАФ one-liners with no branching are covered by higher-level snapshot tests.
- Run `npm test` before every commit. Run `npx vitest -u` after intentional output changes to update snapshots.
