# AGENTS.md

## Mandatory Order of Operations

Before brainstorming, planning, or touching any code:

1. **Get live codebase context via context runes** ‚Äî scope `m` to the modules relevant to the task and `kb` to matching entries; avoid loading all modules unless the task spans the full codebase. Re-run crunes at any point if exploration reveals additional dependencies.
2. **Then brainstorm, plan, and code** ‚Äî in that order.

## Restrictions

- **Never publish to npm directly** ‚Äî publishing is handled by CI on tag push. Only bump the version, update the changelog, commit, tag, and push.
- **Never modify `dist/`** ‚Äî it is generated by `npm run build` (esbuild) and committed only as part of a release. Do not hand-edit it.
- **Never run `npm publish`** unless the user explicitly asks.
- **Always sync `package-lock.json`** ‚Äî after bumping the version in `package.json`, run `npm install` to regenerate the lockfile before committing. The `release` rune will flag a mismatch with a ‚öÝ indicator.
- **Rune files are ESM** ‚Äî all files under `.crunes/runes/` and `src/` use `import`/`export`. Never use `require()`.
- **Local runes run inside `isolated-vm`** ‚Äî they cannot access Node builtins directly. All I/O goes through `utils.fs`, `utils.shell`, `utils.json`, `utils.fetch`, `utils.env`, `utils.archive`, and `utils.cache`.
- **Test before committing rune changes** ‚Äî run `node dist/cli.js use <rune> --plain` to verify output before staging.
- **Fetch module context via `crunes use m=<module>`** for the module(s) you are about to touch ‚Äî fall back to the module's `README.md` only if `crunes` is unavailable.
- **Prefer `crunes` over `Glob`/`Grep`/`ls`** ‚Äî use filesystem tools only when crunes cannot answer the question.
- **Only read files that will change what you implement** ‚Äî ask "will this result change my approach?" before reading anything.

## Build & Development Commands

```bash
# Install dependencies
npm install

# Build the CLI (bundles src/ ‚Üí dist/cli.js via esbuild)
npm run build

# Run all tests
npm test

# Run tests in watch mode
npx vitest

# Update snapshots
npx vitest -u

# Run a built rune against the current project
node dist/cli.js use release --plain
node dist/cli.js list

# Full CI check (mirrors what CI does)
npm test && npm run build && node dist/cli.js --help
```

## Architecture Overview

**Stack**: Node.js ‚â• 20 ‚Ä¢ ESM ‚Ä¢ esbuild (bundler) ‚Ä¢ commander (CLI) ‚Ä¢ isolated-vm (plugin sandboxing) ‚Ä¢ vitest (tests)

**Entry point**: `src/cli/cli.js` ‚Äî process bootstrap, Node snapshot workarounds, and `-v` flag disambiguation.

**Core execution path**: `crunes use <key>` ‚Üí `src/rune/commands/use.js` ‚Üí `src/rune/resolver.js` (`runRune`) ‚Üí resolves local vs. plugin ‚Üí `src/rune/isolation/runner.js` (isolated-vm) ‚Üí returns `Section[]` ‚Üí `src/shared/render.js` ‚Üí stdout.

**Key resolution order** (for a bare key with no prefix):
1. Project config (`.crunes/config.json` ‚Üí `runes.<key>.path`)
2. Enabled plugins (auto-resolved; errors if ambiguous across multiple plugins)

**Prefixes**: `local:<key>` forces project config only. `<plugin>:<key>` forces a specific plugin.

### Module Map

Prefer `crunes use m` ‚Äî this is a static fallback only.

- `cli` ‚Äî Entry point (`cli.js`), Commander setup (`program.js`), `-v` disambiguation, general CLI commands.
- `core` ‚Äî `loadConfig()`, `CircularRuneError`, shared config types.
- `job` ‚Äî Background job tracking: create, list, kill, GC, prefix-match resolution.
- `marketplace` ‚Äî Marketplace source URL management, plugin index browsing and caching.
- `plugin` ‚Äî Plugin registry, install, consent, dep resolution, store paths.
- `project` ‚Äî Reverse-lookup index: hashed project key ‚Üí project directory path.
- `rune` ‚Äî Key resolution, sandboxed VM execution, utils API, permissions.
- `shared` ‚Äî `render.js`, `output.js` ‚Äî cross-cutting utilities with no domain coupling.
- `store` ‚Äî Centralised path helpers for `~/.crunes/` (or `$CRUNES_STORE`).
- `cache` ‚Äî Rune-visible cache store: named key/value buckets backed by JSON files.
- `sqlite` ‚Äî Rune-visible SQLite store: named databases in a central index.
- `template` ‚Äî Rune template listing and scaffolding.

### Module Documentation

Every module has a `README.md` stub at `src/<module>/README.md` (file index, submodules, related modules). **Do not put narrative docs in READMEs** ‚Äî narrative belongs in the KB.

KB vault: `docs/knowledge-base/` ‚Äî module notes at `modules/<module>.md`, flows at `flows/<flow>.md`, system at `system/`.

- **Single module:** `crunes -p use m <module> + kb -m <module>`
- **Submodule:** `crunes -p use m <module>.<submodule>` (dot = submodule path separator, e.g. `m rune.isolation`)
- **Multiple modules:** `crunes -p use m <mod1> <mod2> + kb -m <mod1> <mod2>`
- **Priority order:** chain `+` groups so the most task-relevant module comes first ‚Äî e.g. `crunes -p use m rune.isolation + kb -m rune.isolation + m plugin + kb -m plugin`

Include `kb` alongside `m` when design decisions or subtle gotchas are likely to matter ‚Äî new features, cross-module contracts, anything non-obvious from source. Skip for purely structural reads.

### Context Runes

- `release` ‚Äî Current version, lockfile sync status, branch/tag, last 10 commits, recent changelog, release process steps.
- `m` ‚Äî Module structure tree + file tree. No args: all modules. `m rune` ‚Üí rune module. `m rune.isolation` ‚Üí submodule (dot = path separator). `m rune plugin` ‚Üí multiple modules (space-separated). Sections: `layout` (submodule tree), `files` (JS file tree), `readme`.
- `kb` ‚Äî Knowledge base notes. No args/flags: full index. `-m <name...>` ‚Üí module entries (default when names given). `-f <name...>` ‚Üí flow entries. `-s <name...>` ‚Üí system entries. Sections: one per entry name.

**Fallback** (crunes unavailable): Module Map above for `m`; `docs/knowledge-base/index.md` for `kb`.

## Release Process

1. Bump `version` in `package.json`
2. Run `npm install` to sync `package-lock.json`
3. Add `## [x.y.z] - YYYY-MM-DD` entry to `CHANGELOG.md`
4. Commit: `git commit -m "chore: release vX.Y.Z"`
5. Tag: `git tag vX.Y.Z`
6. Push: `git push origin main --tags` ‚Äî publish CI fires automatically on the tag

Verify before pushing: `node dist/cli.js use release --plain` should show matching versions, `‚úì` on lockfile, and `0 (in sync)` on unpushed.

## Testing Philosophy

- Test **API contracts**, not implementation details.
- Use **prose assertions** for distinct behavioural contracts (error types, option semantics, edge cases).
- Use **snapshots** for scaffold/template string outputs ‚Äî prose tests guard logic, snapshots guard exact formatting.
- Avoid testing **trivial wrappers** ‚Äî one-liners with no branching are covered by higher-level snapshot tests.
- Run `npm test` before every commit. Run `npx vitest -u` after intentional output changes to update snapshots.
